# Architecture

OneLens is a three-piece system: an IntelliJ plugin that extracts
facts via PSI, a Python CLI that imports those facts into a graph
database and embeds them into a vector database, and a Claude Code
skill that bridges natural-language questions to both.

## System at a glance

```
┌── IntelliJ IDE ──────────────────────────────────────┐
│  OneLens Plugin (Kotlin)                             │
│  ├── Collectors (PSI APIs) — classes, members,       │
│  │   call graph, inheritance, Spring, endpoints,     │
│  │   annotations, modules                            │
│  ├── Tools → OneLens → Sync / Install Skill / Toggle │
│  ├── VFS listener → debounced delta export (5 s)     │
│  └── Shells out to onelens CLI                       │
└────────────────────────┬─────────────────────────────┘
                         │ JSON (full or delta)
┌────────────────────────┴─────────────────────────────┐
│  onelens CLI (Python, cyclopts-generated)            │
│  ├── import-graph — auto-detects full vs delta       │
│  ├── retrieve / impact / trace / search / query      │
│  ├── daemon — warm Qwen3 + mxbai                     │
│  └── mcp-server — same ops via MCP stdio             │
└───────────┬──────────────────────────┬───────────────┘
            │                          │
    ┌───────┴────────┐         ┌───────┴────────────┐
    │ FalkorDB       │         │ ChromaDB           │
    │ Cypher + FTS   │         │ Qwen3 embeddings   │
    │ + browser UI   │         │ + mxbai rerank     │
    └────────────────┘         └────────────────────┘
```

## The seams

Every extension point:

1. **Graph backend.** `--backend falkordb | falkordblite | neo4j`.
   Interface in `graph/db.py`. Adding a new backend = one subclass.
2. **Embedder / reranker.** Swap classes in `context/embedder.py` /
   `context/reranker.py`. Config is loaded from pyproject extras.
3. **Collectors.** Add a new `*Collector.kt` under
   `plugin/.../collectors/`, register it in `ExportService`. PSI
   read happens once per sync, so collectors add minimal cost.
4. **MCP tools.** Add `@mcp.tool` in `mcp_server.py`; regenerate
   CLI via `fastmcp generate-cli`. Agents and CLI stay in lockstep.

## Data flows

### Full sync (first time or "Sync Graph")

1. User clicks Tools → OneLens → Sync Graph.
2. `ExportService` runs collectors inside a single ReadAction.
3. JSON written to `~/.onelens/exports/<project>-full-<ts>.json`.
4. Plugin TCP-probes FalkorDB on `localhost:17532` (preflight).
5. Plugin invokes `onelens import-graph <json> --graph <name>
   --context --clear`.
6. CLI detects full type from JSON header, runs `GraphLoader` bulk
   UNWIND, then `CodeMiner.mine` (embedding pass, ~20 min on a
   74K-method project), then PageRank prebake.

### Delta sync (file save)

1. VFS listener catches `.java` save.
2. Debounce window (5 s). Multiple saves collapse into one delta.
3. `DeltaExportService` exports only changed classes + their
   dependencies.
4. Plugin invokes `onelens import-graph <delta.json> --graph
   <name> --context` (no `--clear`).
5. CLI detects delta from JSON header, calls
   `DeltaLoader.apply_delta` which:
   - UNWINDs node/edge upserts,
   - calls `CodeMiner.delete_by_ids` + `delete_methods_of_classes`
     for removed entities (ID-prefix cascade),
   - calls `mine_upserts` for changed methods/classes,
6. End-to-end: < 5 s on a single-file save.

### Natural-language query

1. User asks Claude Code: "what calls UserService?"
2. Skill at `~/.claude/skills/onelens/SKILL.md` routes via Bash:
   `onelens retrieve --query "..." --graph <name>`.
3. `hybrid_retrieve` checks router for exact-name match → direct
   Cypher; else runs FTS + semantic in parallel, RRF-fuses, applies
   PageRank boost, cross-encoder reranks, threshold-filters.
4. Results returned as a structured JSON table; Claude Code
   narrates.

## Open questions

1. Multi-project graphs in one FalkorDB instance — how to scope
   cross-project queries? Currently one graph per project.
2. Incremental PageRank on delta imports — today we rerun on every
   delta, which is O(|V| + |E|). For very large graphs (>1M edges)
   this may need to become incremental.
3. Cross-language (Kotlin + Java mixed projects) — collectors are
   Java-focused; Kotlin PSI works but annotations and Spring
   semantics in Kotlin need explicit testing.
